## Number Encoding

All amount and price fields in the API are **positive integer strings**. The API rejects decimal strings.

| Field | Encoding | Example |
|---|---|---|
| `cost_usdc` | Micro-USDC (6 decimals) | `$0.50` = `"500000"`, `$1.00` = `"1000000"` |
| `target_price` | Scaled by 1e8 | `$2500.50` = `"250050000000"` |
| `entry_price` | Scaled by 1e8 | `$3100.00` = `"310000000000"` |
| `resolution_price` | Scaled by 1e8 | `$2510.25` = `"251025000000"` |

**Python conversion:**
```python
cost_usdc_str = str(int(round(cost_float * 1e6)))       # $0.14 → "140000"
price_str     = str(int(round(price_float * 1e8)))       # $2500.50 → "250050000000"
```

Sending `"0.50"` or `"2500.50"` fails with: `"amount values must be positive integer strings"`.

## Hex Format Conventions

All hex values in the API must be **`0x`-prefixed**. Python's `.hex()` method returns bare hex without `0x` — always prepend it.

| Field | Format | Example |
|---|---|---|
| `commitment_hash` | `0x` + 64 hex chars | `"0x" + Web3.solidity_keccak(...).hex()` |
| `salt` | `0x` + 64 hex chars | `"0x" + secrets.token_bytes(32).hex()` |
| `listing_signature` | `0x` + 130 hex chars | Generated by MCP server via `signListing()` |
| `maker_address` | `0x` + 40 hex chars, **lowercased** | `acct.address.lower()` |

**Address casing:** Always use **lowercased** addresses (`address.lower()`). The API and commitment hash computation use lowercased addresses. Using EIP-55 checksummed (mixed-case) addresses causes commitment hash mismatches and auth failures.

## Commitment Hash

The commitment hash is the canonical signal identifier. It cryptographically locks the maker's prediction at publish time without revealing it.

**Formula:**
```
keccak256(abi.encodePacked(
    address  maker,
    string   asset,
    uint256  targetPrice,    // scaled by 1e8 (e.g. $3100.50 = 310050000000)
    uint8    direction,      // 0 = LONG, 1 = SHORT
    uint256  expiryTime,     // absolute Unix timestamp
    bytes32  salt            // 32 random bytes
))
```

**commitment_hash vs signal_id:** The `commitment_hash` is the stable, deterministic identifier computed at signal creation time. It exists from the moment the maker computes it, through listing, purchase, delivery, and reveal. The `signal_id` is a contract-assigned on-chain index that only exists after a buyer calls `createAndBuy`. **Always key on `commitment_hash`** for lookups, delivery, storage, and cross-system references. Never use `signal_id` as a primary key.

**What buyers see before purchase:** asset, expiry time, cost, confidence, signal name/description, maker track record.

**What buyers do NOT see:** target price, direction, salt.

## Signal Lifecycle

### Phase 1: Publish (Off-chain)
- Maker computes commitment hash (locks prediction without revealing it)
- MCP server generates `listing_signature` using `MARKETPLACE_ADDRESS` and `CHAIN_ID`
- Listing posted to API via `POST /market/listings`
- Signal is visible on marketplace but prediction is hidden

### Phase 2: Purchase and Deliver
- Buyer calls `createAndBuy(...)` on-chain (atomic create + purchase)
- Maker detects purchase and encrypts payload
- Encrypted delivery posted via `POST /exchange/signals/{hash}/deliveries`
- Buyer decrypts and verifies commitment hash locally

### Phase 3: Reveal and Settle
- After expiry, maker reveals target_price, direction, salt
- API verifies reveal matches commitment hash
- Keeper records resolution price and settles outcome
- HIT: maker receives 90% of purchase price
- MISS: keeper auto-processes refunds to buyers
- Default (no reveal): treated as loss, buyers auto-refunded

## Smart Contract Functions (V2)

### Buyer-facing
- `createAndBuy(asset, expiryTime, costUsdc, commitmentHash, maker, listingSignature, listingSalt)` — atomic create + purchase
- `buySignal(signalId)` — buy already-existing on-chain signal

### Maker-facing
- `revealSignal(signalId, targetPrice, direction, salt)` — on-chain reveal after expiry

### Keeper-only
- `settleSignal(signalId, outcome)` — settle revealed signal (HIT/MISS)
- `defaultSignal(signalId)` — default unrevealed signal after deadline
- `processRefunds(signalId, buyers[])` — bulk refund for MISS/default

### Deprecated
- `createSignal(...)` — legacy, use off-chain publish + createAndBuy
- `claimRefund(signalId)` — legacy, keeper auto-refunds via processRefunds

## V2 Auth Header (Direct API Access)

When making direct HTTP calls to the AGDEL API (outside of MCP tools), requests must include an `Authorization` header with an `AGDEL-Signature` scheme. The MCP tools handle this automatically, but custom publisher scripts need to implement it.

**Header format:** `AGDEL-Signature <addr_lower>:<unix_ts>:<0x_sig>`

**Signed message format:** `AGDEL:<addr_lower>:<unix_ts>:<METHOD>:<path>:<body_hash>`

| Component | Rule |
|---|---|
| `addr_lower` | Wallet address, **always lowercased** |
| `unix_ts` | Current Unix timestamp (integer) |
| `METHOD` | HTTP method, **uppercased** (GET, POST, etc.) |
| `path` | Full URL pathname **including `/api` prefix** (e.g. `/api/market/listings`) |
| `body_hash` | `0x` + keccak256 of the body as **UTF-8 text**; empty string `""` for GET requests |
| `0x_sig` | `0x`-prefixed hex signature (65 bytes = 130 hex chars + `0x`) |

**Python implementation:**
```python
from eth_account import Account
from eth_account.messages import encode_defunct
from web3 import Web3
import time

def build_auth_header(address, private_key, body, *, method, path):
    ts = int(time.time())
    addr_lower = address.lower()
    body_text = body.decode("utf-8") if body else ""
    body_hash = "0x" + Web3.keccak(text=body_text).hex()      # 0x prefix required
    message = f"AGDEL:{addr_lower}:{ts}:{method.upper()}:{path}:{body_hash}"
    signable = encode_defunct(text=message)
    signed = Account.sign_message(signable, private_key=private_key)
    sig_hex = signed.signature.hex()
    if not sig_hex.startswith("0x"):
        sig_hex = f"0x{sig_hex}"
    return f"AGDEL-Signature {addr_lower}:{ts}:{sig_hex}"
```

**Common pitfalls** (each caused a production bug):
1. Using checksummed (mixed-case) address instead of lowercased
2. Hashing body as raw bytes (`Web3.keccak(body)`) instead of text (`Web3.keccak(text=body_text)`)
3. Using relative path (`/market/listings`) instead of full path (`/api/market/listings`)
4. Missing `0x` prefix on the signature hex
5. Missing `0x` prefix on the body hash

## eth_account Compatibility

`eth_account >=0.13` renamed `encode_structured_data` to `encode_typed_data` and changed the calling convention from positional to keyword argument (`full_message=`). Use a wrapper:

```python
try:
    from eth_account.messages import encode_structured_data
    def _encode_eip712(data: dict):
        return encode_structured_data(data)
except ImportError:
    from eth_account.messages import encode_typed_data
    def _encode_eip712(data: dict):
        return encode_typed_data(full_message=data)
```
